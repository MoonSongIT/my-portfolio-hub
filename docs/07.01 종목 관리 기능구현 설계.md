# 종목 관리 기능 구현 설계 (Master DB)

> **목적**: 설정 화면에서 **국내/해외** 주식·ETF 종목 마스터 DB를 일괄 다운로드하고,
> 추가/변경분을 증분 업데이트(upsert)하며, 이를 종목 탐색·관심종목·AI 분석 등 앱 전역에서
> "마스터 조회 소스"로 사용할 수 있도록 하는 기능의 전체 설계 문서입니다.

## 1. 요구사항 (Requirements)

### 1.1 사용자 요구 (원문 반영)

- 설정 화면에서 **주식 종목 DB 관리** 진입
- **국내 주식**: 코스피(KOSPI), 코스닥(KOSDAQ), **NXT**(Nextrade 대체거래소), **ETF**(KRX 상장)
- **해외 주식**: **US** 공통, **NYSE**(뉴욕), **NASDAQ**(나스닥) — 및 미국 ETF
- 상장시장 구분을 포함한 **마스터 DB** 일괄 다운로드
- 추가/변경된 종목은 **update(upsert)** 방식으로 반영 (전체 삭제 후 재삽입 X)
- 관리된 마스터 데이터는 **종목 탐색**, 관심종목 추가, AI 분석의 티커 추출 등
  앱 내 다른 기능에서 단일 소스로 재사용

### 1.2 파생 비기능 요구

| 항목         | 요구                                                          |
|--------------|---------------------------------------------------------------|
| 응답성       | 초기 전체 다운로드 ≤ 60s (국내 + 해외 합산), 증분 ≤ 10s       |
| 저장 용량    | 국내 ~3,500 + ETF ~800 + 해외 ~1,000 = 약 5,000~8,000 row     |
| 오프라인 조회| 다운로드 이후에는 네트워크 없이도 탐색·선택 가능 (IndexedDB)  |
| 쓰기 무결성  | 다운로드 중 실패해도 기존 데이터 보존 (트랜잭션 기반 upsert)  |
| 관측성       | 시장별 수집 개수·에러·소요시간 UI 노출                         |
| 중복 방지    | `(market, ticker)` 복합 유니크 키로 중복 방지                 |

### 1.3 명시적 비요구 (Out of Scope, 이번 설계)

- 실시간 시세 연동 (마스터 DB는 "종목 메타"만 보관, 시세는 기존 `stockApi`/`naverApi` 담당)
- 재무·공시·뉴스 연동 (`06.01 AI 종목 종합분석` 설계에서 다룸)
- Supabase 동기화 (Phase 6+ 에서 별도 설계)
- 종목 스크리너(필터링 대시보드)

## 2. 도메인 모델 (Schema)

### 2.1 통합 스키마 — `stock_master`

모든 시장 종목을 **단일 테이블**에 저장하고, `category` + `exchange` 두 축으로 구분합니다.

```ts
interface StockMasterRow {
  // ── 식별자 ───────────────────────────────
  id:         string      // `${category}:${exchange}:${ticker}` 복합 PK
  ticker:     string      // 순수 티커 (예: "005930", "AAPL")
  name:       string      // 표시명 (한글명 우선, 영문 fallback)
  nameEn?:    string      // 영문명 (한국 종목에만 의미)

  // ── 시장 분류 (핵심) ─────────────────────
  category:   'DOMESTIC' | 'OVERSEAS'         // 국내 / 해외 (1차 축)
  exchange:   'KOSPI' | 'KOSDAQ' | 'NXT' | 'KRX_ETF'
            | 'NYSE'  | 'NASDAQ' | 'AMEX' | 'US_ETF'
  type:       'EQUITY' | 'ETF' | 'ETN' | 'REIT' | 'PREFERRED'
  country:    'KR' | 'US'
  currency:   'KRW' | 'USD'

  // ── 부가 정보 ────────────────────────────
  sector?:    string      // 업종 (가능한 시장만)
  industry?:  string
  isin?:      string      // 국제증권식별번호 (KRX 공식)
  corpCode?:  string      // DART 고유 corp_code (8자리) — 한국 종목 전용

  // ── 운영 필드 ────────────────────────────
  isCustom:   boolean     // 사용자가 수동 추가한 종목
  isActive:   boolean     // 상장폐지 감지용 (기본 true)
  firstSeenAt:string      // 최초 수집 시각 (ISO)
  updatedAt:  string      // 마지막 갱신 시각 (ISO)
  source:     'DART' | 'KRX' | 'NAVER' | 'YAHOO' | 'SEC' | 'MANUAL'
}
```

### 2.2 인덱스 설계 (Dexie)

```js
// src/utils/stockMasterDb.js
import Dexie from 'dexie'

export const stockMasterDb = new Dexie('StockMasterDB')

stockMasterDb.version(1).stores({
  // PK = id, 보조 인덱스 5개
  stocks:
    'id, ' +
    '[category+exchange], ' +  // 시장별 목록 조회
    '[category+ticker], ' +    // 시장 안에서 티커 검색
    'ticker, ' +               // 전역 티커 조회
    'name, ' +                 // 한글명 LIKE 검색
    'nameEn',                  // 영문명 LIKE 검색

  // 수집 이력 (관측·디버깅용)
  syncLogs:
    '++id, syncedAt, exchange, status',
})
```

### 2.3 왜 IndexedDB(Dexie) 인가

| 기준           | LocalStorage (현행)   | IndexedDB (신규)          |
|----------------|-----------------------|---------------------------|
| 최대 용량      | 5~10 MB (브라우저 별) | 수백 MB~                 |
| 쿼리           | 전량 로드 후 필터     | 인덱스 기반 부분 조회     |
| 업데이트 단위  | 전체 직렬화 재저장    | row 단위 put/delete       |
| 예상 row 수    | 5,000+ 저장 시 체감 ↓ | 수만 row 도 성능 유지     |

→ 국내만 KOSPI(~800) + KOSDAQ(~1,700) + NXT(~800 중복 포함) + ETF(~800) 이미 3,000+.
전체 직렬화 기반 LocalStorage는 쓰기 비용이 커서 **IndexedDB(Dexie)** 로 이전합니다.

## 4. 데이터 소스 매트릭스

| category   | exchange  | 1차 소스                          | 2차 소스 (폴백)            | 갱신 빈도   |
|------------|-----------|-----------------------------------|----------------------------|-------------|
| DOMESTIC   | KOSPI     | DART `list.json` (corp_cls=Y)     | KRX 상장법인목록 (XLS)     | 주 1회       |
| DOMESTIC   | KOSDAQ    | DART `list.json` (corp_cls=K)     | KRX 상장법인목록 (XLS)     | 주 1회       |
| DOMESTIC   | **NXT**   | Nextrade 거래소 고시 CSV/페이지   | KRX `nxt_issue_list`       | 주 1회       |
| DOMESTIC   | KRX_ETF   | 네이버 `etfItemList.nhn`          | KRX ETF 통계 페이지         | 주 1회       |
| OVERSEAS   | NYSE      | Yahoo screener + 폴백 티커        | NASDAQ Trader `nyelisted`  | 주 1회       |
| OVERSEAS   | NASDAQ    | Yahoo screener + 폴백 티커        | NASDAQ Trader `nasdaqlisted`| 주 1회       |
| OVERSEAS   | US_ETF    | Yahoo `quote` + 폴백 티커          | ETFdb(CSV, 수동 업로드)    | 월 1회       |

### 4.1 소스별 쿼리 규칙 상세

#### KOSPI / KOSDAQ (DART)
- 현 구현(`fetchDartListByCls`)은 `pblntf_ty=A` + `89일 × 4윈도우` 로 연 사업보고서를 스캔.
- 장점: API Key 만 있으면 corp_code / 종목코드 / 회사명 전부 획득.
- 제약: 연 1회 보고서 미제출 종목(신규 상장 직후 등)은 누락 가능 → **KRX 상장법인 XLS** 로 2차 보강.

#### NXT (신규)
- Nextrade(넥스트레이드)는 2025년 오픈한 국내 대체거래소(ATS).
- NXT는 **별도 상장**이 아니라 **KOSPI/KOSDAQ 종목의 NXT 동시 거래 허용** 형태.
- 따라서 `exchange=NXT` 는 **"NXT 거래 가능 종목"** 을 의미하며, 보통 KOSPI/KOSDAQ row와 중복되지 않도록 **거래 가능 플래그**를 추가하는 모델이 정확합니다.
- **모델 변경 제안**:

  ```ts
  // exchange 를 단일 문자열 대신 배열로 확장 (플래그 모델)
  tradableOn: ('KOSPI' | 'KOSDAQ' | 'NXT')[]   // DOMESTIC 전용
  primaryExchange: 'KOSPI' | 'KOSDAQ'           // 원 상장 거래소
  ```

  단, 설정 UI는 사용자 요구 그대로 "코스피 / 코스닥 / NXT / ETF" 4개 카드로 표시하되,
  내부적으로는 `tradableOn` 배열에 NXT 포함 여부로 분류합니다.

#### KRX ETF
- 기존 `fetchNaverEtfList()` 로직 유지 (sosok=0/1 페이징).

#### NYSE / NASDAQ
- 기존 `fetchYahooStocks()` 의 **screener + 폴백 티커** 방식은 커버리지가 제한적(~150종목).
- 본 설계는 **NASDAQ Trader 공식 파일** 을 2차 소스로 추가:
  - `https://ftp.nasdaqtrader.com/dynamic/SymDir/nasdaqlisted.txt`
  - `https://ftp.nasdaqtrader.com/dynamic/SymDir/otherlisted.txt` (NYSE/AMEX 포함)
  - 파이프(`|`) 구분 텍스트 → 헤더/Test Issue 제거 후 전체 티커 수집 (~8,000개).
- Yahoo는 `sector`, `shortName` 보강용으로만 사용 (옵션).

#### US ETF
- Yahoo는 ETF 리스트 전용 screener 가 빈약 → 하드코딩 50~70종 + (선택) ETFdb CSV 업로드.

### 5.4 정규화 규칙 (`normalize.js`)

```js
// 예: DART → StockMasterRow
export function fromDart(item, corpCls) {
  return {
    id:         `DOMESTIC:${corpCls === 'Y' ? 'KOSPI' : 'KOSDAQ'}:${item.stock_code}`,
    ticker:     item.stock_code,
    name:       item.corp_name.trim(),
    category:   'DOMESTIC',

    type:       'EQUITY',
    country:    'KR',
    currency:   'KRW',
    corpCode:   item.corp_code,
    tradableOn: [corpCls === 'Y' ? 'KOSPI' : 'KOSDAQ'],  // NXT 는 nxt.js 가 병합
    source:     'DART',
    isCustom:   false,
    isActive:   true,
  }
}
```

## 6. 증분 업데이트(Diff & Upsert) 알고리즘

사용자 요구 "추가된 정보는 update 방식"의 실제 구현:

```
┌─────────────────────────────────────────────────────────────┐
│ 1. 서버에서 exchange 별 최신 목록(remote) 수집               │
│ 2. IndexedDB 에서 해당 exchange 의 기존 목록(local) 조회      │
│ 3. Set 연산:                                                 │
│     added   = remote.keys - local.keys                      │
│     removed = local.keys  - remote.keys                     │
│     common  = remote.keys ∩ local.keys                      │
│ 4. common 중 name/sector/corpCode 등이 달라진 것 = changed   │
│ 5. Dexie 트랜잭션 안에서:                                     │
│     · added   → put(neRow)                                  │
│     · changed → put(mergedRow)  // firstSeenAt 유지          │
│     · removed → update(isActive=false)  // 삭제 대신 soft    │
│ 6. syncLogs 에 결과 append                                    │
└─────────────────────────────────────────────────────────────┘
```

### 6.1 Soft Delete 의 이유

- 사용자가 과거에 매매했던 상장폐지 종목은 **일지 조회/과거 수익률 계산**에서 여전히 필요.
- 탐색 UI에서는 기본적으로 `isActive = true` 만 노출, "상폐 포함 검색" 토글 제공.

### 6.2 병합 규칙 (changed row)

```js
const merged = {
  ...localRow,           // firstSeenAt, isCustom(사용자 편집분 존중)
  ...remoteRow,          // 서버 최신 필드 덮어쓰기
  firstSeenAt: localRow.firstSeenAt,  // 최초 감지일은 보존
  updatedAt:   new Date().toISOString(),
  // isCustom 이 true 였다면 강제로 보존 (서버가 덮어쓰지 않도록)
  isCustom:    localRow.isCustom || remoteRow.isCustom,
}
```

### 6.3 트랜잭션 경계

```js
await stockMasterDb.transaction('rw', stockMasterDb.stocks, async () => {
  await stockMasterDb.stocks.bulkPut([...added, ...changed])
  await stockMasterDb.stocks
    .where('id').anyOf(removedIds)
    .modify({ isActive: false, updatedAt: now })
})
```

- 부분 실패 시 롤백되어 기존 DB 일관성 유지.
- Dexie `bulkPut` 은 내부적으로 한 번의 IDB 트랜잭션으로 처리 → 수천 row 도 1~2초.

### 7.1 `src/utils/stockMasterDb.js` (Dexie 래퍼)

```js
export const stockMasterDb = new Dexie('StockMasterDB')
stockMasterDb.version(1).stores({ /* §2.2 */ })

export async function upsertSync(exchange, remoteRows) {
  const local = await stockMasterDb.stocks
    .where('[category+exchange]').equals([categoryOf(exchange), exchange]).toArray()

  const localById  = new Map(local.map(r => [r.id, r]))
  const remoteById = new Map(remoteRows.map(r => [r.id, r]))

  const added   = remoteRows.filter(r => !localById.has(r.id))
  const removed = local.filter(r => !remoteById.has(r.id) && !r.isCustom)
  const changed = remoteRows
    .filter(r => localById.has(r.id))
    .map(r => mergeRow(localById.get(r.id), r))
    .filter(m => m.__dirty)

  await stockMasterDb.transaction('rw', stockMasterDb.stocks, async () => {
    if (added.length)   await stockMasterDb.stocks.bulkPut(added)
    if (changed.length) await stockMasterDb.stocks.bulkPut(changed)
    if (removed.length) await stockMasterDb.stocks
      .where('id').anyOf(removed.map(r => r.id))
      .modify({ isActive: false, updatedAt: new Date().toISOString() })
  })

  return { added: added.length, changed: changed.length, removed: removed.length }
}

export async function searchByQuery(query, { limit = 20, includeInactive = false } = {}) {
  const q = query.trim().toLowerCase()
  if (!q) return []

  // 1) ticker 전방 일치 (가장 빠름)
  const byTicker = await stockMasterDb.stocks
    .where('ticker').startsWithIgnoreCase(q).limit(limit).toArray()

  // 2) 한글명/영문명 부분 일치 (Collection.filter 사용)
  const byName = await stockMasterDb.stocks
    .filter(r => r.name.toLowerCase().includes(q) || r.nameEn?.toLowerCase().includes(q))
    .limit(limit).toArray()

  const merged = dedupeById([...byTicker, ...byName])
  return includeInactive ? merged : merged.filter(r => r.isActive)
}
```

### 7.2 `src/api/stockMasterApi.js`

```js
export async function fetchExchange(exchange) {
  const res = await fetch(`/api/stock-master?exchange=${exchange}`, {
    signal: AbortSignal.timeout(30_000),
  })
  if (!res.ok) throw new Error(`stock-master ${exchange} HTTP ${res.status}`)
  return res.json()  // { rows, count, errors, collectedAt }
}

export const EXCHANGES = {
  DOMESTIC: ['KOSPI', 'KOSDAQ', 'NXT', 'KRX_ETF'],
  OVERSEAS: ['NYSE',  'NASDAQ', 'US_ETF'],
}

export async function syncCategory(category, { onProgress, mode = 'upsert' } = {}) {
  const exchanges = EXCHANGES[category] || []
  const stats = {}
  for (let i = 0; i < exchanges.length; i++) {
    const ex = exchanges[i]
    onProgress?.({ phase: 'fetch', exchange: ex, done: i, total: exchanges.length })
    try {
      const { rows } = await fetchExchange(ex)
      onProgress?.({ phase: 'upsert', exchange: ex, done: i, total: exchanges.length })
      stats[ex] = await upsertSync(ex, rows)
    } catch (err) {
      stats[ex] = { error: err.message }
    }
  }
  onProgress?.({ phase: 'done', done: exchanges.length, total: exchanges.length })
  return stats
}

export async function syncAll(opts) {
  const domestic = await syncCategory('DOMESTIC', opts)
  const overseas = await syncCategory('OVERSEAS', opts)
  return { DOMESTIC: domestic, OVERSEAS: overseas }
}
```
