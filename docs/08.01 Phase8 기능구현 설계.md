## Feature 1 — 차트 동적 데이터 로딩

### 현재 상태 진단

`src/components/charts/CandlestickChart.jsx`:
- 부모(`StockDetail.jsx`)에서 `range` 파라미터 (예: `'6mo'`)로 한 번만 fetch
- lightweight-charts 내장 zoom(스크롤)은 기존 데이터 범위 내에서만 동작
- 사용자가 차트를 왼쪽으로 드래그하거나 스케일을 줄이면 데이터 없는 빈 공간 노출
- timeframe 버튼(1D/1W/1M)은 기존 데이터를 resample할 뿐, 추가 fetch 없음

### 해결 방법 — 가시 범위 감지 + 점진적 fetch

#### lightweight-charts API 활용

```js
// 가시 범위 변경 감지
chart.timeScale().subscribeVisibleLogicalRangeChange((range) => {
  if (range.from < THRESHOLD) {
    // 데이터가 부족 → 더 긴 범위 fetch 요청
    fetchMoreHistory()
  }
})
```

#### 데이터 로딩 전략

```
현재:
  StockDetail(range='6mo') → fetchHistory() → CandlestickChart(data=고정)

변경:
  StockDetail
    ├─ loadedRange 상태 관리 (현재 로드된 가장 긴 기간)
    ├─ handleNeedMoreData(earliestDate) 콜백
    └─ 병합된 history prop 전달

  CandlestickChart
    ├─ subscribeVisibleLogicalRangeChange 이벤트 구독
    ├─ 왼쪽 경계 도달 감지 (전체 데이터의 10% 지점)
    └─ onNeedMoreData(earliestDate) 콜백 호출
```

#### Range 확장 단계

| 현재 로드 범위 | 다음 fetch 범위 | 비고 |
|-------------|--------------|------|
| 1mo 이하 | 6mo | 소폭 확장 |
| 6mo 이하 | 1y | 중간 확장 |
| 1y 이하 | 2y | 대폭 확장 |
| 2y 이하 | 5y | 최대 |
| 5y 초과 | fetch 중단 | 최대 범위 도달 |

#### 핵심 구현 포인트

1. **디바운스 300ms** — 스크롤 이벤트가 빠르게 연속 발생하므로 연속 fetch 방지
2. **로딩 인디케이터** — fetch 중 차트 좌측 상단에 작은 스피너 표시
3. **중복 제거 병합** — 새 데이터 + 기존 데이터 날짜 기준 merge → sort → dedup
4. **최대 범위 제한** — 5년 초과 시 더 이상 fetch하지 않음 (API 한계 + UX)
5. **오프라인 처리** — fetch 실패 시 기존 데이터 유지, toast 에러 표시

## Feature 2 — 대시보드 수익률 추이 차트

### 현재 상태 진단

```
문제:
  ProfitLineChart → samplePriceHistory (하드코딩된 샘플) 사용

실제 데이터는 이미 존재:
  dailyPnlStore.snapshots:
    key: 'ticker|date|accountId'
    value: {
      ticker, date, accountId, name, market,
      closePrice, quantity, avgBuyPrice,
      dailyPnl, dailyPnlRate,
      cumulativePnl, cumulativePnlRate
    }

Dashboard.jsx:
  useDailyPnlStore import는 있지만 ProfitLineChart에 전달하지 않음
```

### 해결 방법 — dailyPnlStore 집계 → 포트폴리오 수준 수익률

#### 집계 로직

```
날짜 D에 대한 포트폴리오 수익률:
  총 투자금(D) = Σ (보유 종목별 avgBuyPrice × quantity)
  총 평가금(D) = Σ (보유 종목별 closePrice_D × quantity)
  수익률(D, %) = (총평가금 - 총투자금) / 총투자금 × 100
```

#### 처리 흐름

```
1. 앱 기동 시 (이미 구현됨)
   dailyPnlStore.loadFromDB() → 최근 180일 IDB에서 로드

2. Dashboard 진입 시
   a. getCumulativeData() 호출 → per-ticker 스냅샷 배열
   b. aggregatePortfolioHistory(snapshots, period) 집계
   c. 날짜별 groupBy → 종목 합산
   d. USD 보유종목 → exchangeRate(KRW) 변환 후 합산
   e. ProfitLineChart에 집계 결과 전달

3. 데이터 없는 경우
   a. snapshotToday() 자동 실행 (오늘치 최초 스냅샷 생성)
   b. 1건이면 라인 대신 현재 수익률 카드만 표시
   c. 0건이면 EmptyState ("아직 투자 기록이 없습니다")
```

#### ProfitLineChart 데이터 포맷

```js
// 신규 집계 결과 포맷
[{
  date: '2026-04-25',          // YYYY-MM-DD
  totalValue: 52300000,        // 총 평가금액 (KRW)
  investedValue: 48000000,     // 총 투자금액 (KRW)
  returnRate: 6.87,            // 누적 수익률 (%)
  dailyReturn: 0.23,           // 당일 수익률 (%)
}]
```

#### 빈 화면 대신 단계별 UX

| 스냅샷 수 | 표시 방식 |
|--------|---------|
| 0건 | EmptyState: "아직 투자 기록이 없습니다" + 매매일지 작성 유도 |
| 1건 (오늘만) | 현재 수익률 수치 카드 + "기록이 쌓이면 추이가 표시됩니다" |
| 2건 이상 | 정상 라인차트 |


## 기술 리스크 & 대응

| 리스크 | 영향 | 대응 |
|--------|------|------|
| Yahoo/Naver API 2년치 이상 데이터 미제공 | 5y 범위 fetch 불가 | 최대 fetch 가능 범위 감지 후 isMaxRange 플래그 |
| lightweight-charts 가시 범위 이벤트 과다 | fetch 폭주 | 300ms 디바운스 + isFetching 가드 |
| dailyPnlStore 스냅샷 과거 누락 | 수익률 추이 빈 구간 | 마지막 유효값 forward-fill |
| 포트폴리오에 KRW/USD 혼용 | 수익률 집계 오류 | exchangeRate로 USD→KRW 변환 후 합산 |
| 신규 사용자 스냅샷 0건 | 빈 차트 | EmptyState + snapshotToday() 자동 실행 |

## UI/UX 방향

### Feature 1 — 차트 동적 로딩
- 추가 fetch 중: 차트 좌측 상단 작은 `로딩 중...` 스피너
- 최대 범위 도달: 추가 드래그해도 조용히 처리 (에러 없음)
- fetch 실패: `데이터를 불러오지 못했습니다` toast + 기존 데이터 유지

### Feature 2 — 수익률 추이
- 기간 토글: 7일 / 30일 / 90일
- 수익 구간 초록, 손실 구간 빨강 배경 채우기
- 마지막 업데이트 시간 표시 (`오늘 오후 3:30 기준`)
